# 治理数据模型设计稿：SM1

## 核心判断
`scout-media-agents` 的职责不是直接“多平台乱抓”，而是维护一套可审阅、可计划、可追溯的查询治理账本。

## 主链路
```text
topic -> seed -> expansion_candidate -> approved_query_unit -> schedule_state -> crawl_run -> export_job
```

## 对象定义
### Topic
- 语义: 一个长期追踪方向，例如 `AI agents`、`中东战况`
- 核心字段:
  - `id`
  - `name`
  - `description`
  - `status`
  - `priority`
  - `platforms`
  - `owner`

### Seed
- 语义: topic 的原始查询种子
- 核心字段:
  - `id`
  - `topicId`
  - `keyword`
  - `locale`
  - `source`
  - `status`

### ExpansionCandidate
- 语义: 由规则或 LLM 扩出来的候选词
- 核心字段:
  - `id`
  - `seedId`
  - `candidate`
  - `sourceType`
  - `reviewStatus`
  - `evidence`
  - `score`

### QueryUnit
- 语义: 已批准、可调度执行的查询单元
- 核心字段:
  - `id`
  - `topicId`
  - `seedId`
  - `query`
  - `platform`
  - `reviewedFromExpansionId`
  - `schedule`
  - `enabled`

### CrawlRun
- 语义: 某次实际计划/执行记录
- 核心字段:
  - `id`
  - `queryUnitId`
  - `plannedAt`
  - `startedAt`
  - `endedAt`
  - `status`
  - `recordsCollected`
  - `runner`
  - `error`

## 第一阶段落地策略
1. 先复用现有 `seedRegistry` 与 `queryScheduleState` 的能力，不做大拆。
2. 通过新增 registry/state 文件，把 topic、review、run 三个最急需可观测面补上。
3. CLI 只解决运营最核心的“看列表、批审批、看下一批计划、看 run”问题。
